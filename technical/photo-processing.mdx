# Photo Processing

Ello extracts and processes contact photos from vCard imports. This document covers the photo processing pipeline.

## Overview

When importing contacts with embedded photos:

1. **Extract** base64 photo data from vCard
2. **Decode** to binary image buffer
3. **Process** using Sharp (resize, compress)
4. **Save** to disk in hash-based directory structure
5. **Store** hash reference in database

## Size Variants

Each photo is processed into four size variants:

| Variant | Dimensions | Quality | Use Case |
|---------|------------|---------|----------|
| `thumbnail` | 48x48 px | 80% | Contact list rows |
| `small` | 96x96 px | 82% | Compact cards |
| `medium` | 200x200 px | 85% | Detail view |
| `large` | 400x400 px | 88% | Full view |

## Processing Pipeline

### 1. Base64 Extraction

Photos are embedded in vCards as base64:

```
PHOTO;ENCODING=b;TYPE=JPEG:/9j/4AAQSkZJRg...
```

Or in vCard 4.0 format:

```
PHOTO:data:image/jpeg;base64,/9j/4AAQSkZJRg...
```

### 2. Buffer Conversion

```typescript
const buffer = Buffer.from(base64Data, 'base64');
```

### 3. Sharp Processing

```typescript
import sharp from 'sharp';

await sharp(buffer)
  .rotate()                    // Auto-rotate based on EXIF
  .resize(width, height, {
    fit: 'cover',              // Fill entire area
    position: 'attention'      // Smart crop (focus on faces)
  })
  .jpeg({
    quality: quality,          // 80-88 depending on size
    mozjpeg: true,             // Advanced compression
    progressive: true          // Progressive loading
  })
  .toFile(outputPath);
```

### 4. Processing Options

| Option | Value | Purpose |
|--------|-------|---------|
| `rotate()` | Auto | Fix orientation from EXIF |
| `fit: 'cover'` | - | Fill dimensions, crop excess |
| `position: 'attention'` | - | Smart crop focusing on important areas |
| `mozjpeg: true` | - | Better compression ratio |
| `progressive: true` | - | Faster perceived loading |

## File Storage

### Hash Generation

File names are MD5 hashes of the contact ID:

```typescript
const hash = crypto.createHash('md5')
  .update(contactId.toString())
  .digest('hex');
// Example: "a1b2c3d4e5f67890a1b2c3d4e5f67890"
```

### Directory Structure

Photos are organized by size and hash prefix:

```
photos/
├── thumbnail/
│   ├── a1/
│   │   └── a1b2c3d4e5f67890a1b2c3d4e5f67890.jpg
│   ├── b2/
│   │   └── b2c3d4e5f67890a1b2c3d4e5f67890.jpg
│   └── ...
├── small/
│   └── ...
├── medium/
│   └── ...
└── large/
    └── ...
```

The two-character prefix subdirectory prevents filesystem issues with large numbers of files in a single directory.

### Database Reference

Only the hash is stored in the database:

```sql
UPDATE contacts SET photo_hash = 'a1b2c3d4e5f67890a1b2c3d4e5f67890' WHERE id = 123;
```

## URL Construction

Photo URLs are constructed from the hash:

```typescript
function getPhotoUrl(hash: string, size: string): string {
  const prefix = hash.substring(0, 2);
  return `/photos/${size}/${prefix}/${hash}.jpg`;
}

// Example:
getPhotoUrl('a1b2c3d4e5f67890a1b2c3d4e5f67890', 'thumbnail')
// → "/photos/thumbnail/a1/a1b2c3d4e5f67890a1b2c3d4e5f67890.jpg"
```

## Serving Photos

Photos are served as static files via Fastify:

```typescript
import fastifyStatic from '@fastify/static';

await app.register(fastifyStatic, {
  root: photosPath,
  prefix: '/photos/',
  decorateReply: false
});
```

## File Sizes

Typical file sizes after processing:

| Variant | Approximate Size |
|---------|------------------|
| `thumbnail` | 2-4 KB |
| `small` | 6-10 KB |
| `medium` | 15-25 KB |

Total per contact: ~60-90 KB

| Disk write failure | Log error, skip photo |

## Implementation

### photoProcessor.ts

```typescript
import sharp from 'sharp';
import crypto from 'crypto';
import fs from 'fs/promises';
import path from 'path';

const SIZES = [
  { name: 'thumbnail', width: 48, height: 48, quality: 80 },
  { name: 'small', width: 96, height: 96, quality: 82 },
  { name: 'medium', width: 200, height: 200, quality: 85 },
  { name: 'large', width: 400, height: 400, quality: 88 }
];

export async function processPhoto(
  base64Data: string,
  contactId: number
): Promise<string> {
  const buffer = Buffer.from(base64Data, 'base64');
  const hash = crypto.createHash('md5')
    .update(contactId.toString())
    .digest('hex');
  const prefix = hash.substring(0, 2);

  for (const size of SIZES) {
    const dirPath = path.join(photosPath, size.name, prefix);
    await fs.mkdir(dirPath, { recursive: true });

    await sharp(buffer)
      .rotate()
      .resize(size.width, size.height, {
        fit: 'cover',
        position: 'attention'
      })
      .jpeg({
        quality: size.quality,
        mozjpeg: true,
        progressive: true
      })
      .toFile(path.join(dirPath, `${hash}.jpg`));
  }

  return hash;
}
```

## Optimization Tips

### Batch Processing

Process photos in batches to manage memory:

```typescript
const BATCH_SIZE = 100;
for (let i = 0; i < contacts.length; i += BATCH_SIZE) {
  const batch = contacts.slice(i, i + BATCH_SIZE);
  await Promise.all(batch.map(c => processPhoto(c.photo, c.id)));
  sharp.cache(false); // Clear cache between batches
}
```

### Skip Existing

Don't reprocess existing photos:

```typescript
const exists = await fs.access(photoPath)
  .then(() => true)
  .catch(() => false);

if (!exists) {
  await processPhoto(data, contactId);
}
```

### Parallel Processing

Process sizes in parallel:

```typescript
await Promise.all(SIZES.map(size =>
  sharp(buffer)
    .resize(size.width, size.height, options)
    .toFile(getPath(size.name))
));
```

## Troubleshooting

<AccordionGroup>
  <Accordion title="Sharp installation fails">
    Sharp requires native dependencies:

    **macOS:**
    ```bash
    brew install vips
    ```

    **Linux:**
    ```bash
    apt-get install libvips-dev
    ```
  </Accordion>

  <Accordion title="Out of memory during import">
    - Reduce batch size
    - Lower Sharp concurrency: `sharp.concurrency(1)`
    - Clear cache between operations
    - Process photos after import completes
  </Accordion>

  <Accordion title="Photos not displaying">
    Check:
    1. `PHOTOS_PATH` environment variable
    2. Directory permissions
    3. Static file route configuration
    4. Photo hash in database matches filename
  </Accordion>

  <Accordion title="Corrupted images">
    - Verify source image is valid
    - Check for truncated base64 data
    - Try without mozjpeg: `{ mozjpeg: false }`
  </Accordion>
</AccordionGroup>
